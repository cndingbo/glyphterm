## 为何选择 Apache-2.0（而非 MIT / GPL / AGPL）

### 1. 产品定位

GlyphTerm 是**可嵌入、可扩展**的终端与工作区基础设施（Rust crate + 桌面壳）。预期场景包括：

- 个人与团队日常使用
- 企业内部定制或二次集成
- 与 AI、远程、预览等可选模块组合

Apache-2.0 在「开放」与「可集成」之间平衡最好，且比 MIT 多一层**明示专利授权**，对可能被大量复用的基础组件更合适。

### 2. 与各备选对比

| 许可 | 优点 | 对 GlyphTerm 的顾虑 | 结论 |
|------|------|---------------------|------|
| **MIT** | 极简、生态熟悉 | 无专利条款；大型组织法务常更偏好 Apache | 不选为唯一许可 |
| **Apache-2.0** | 专利授权、贡献者友好、企业常用 | 文件略长 | **采用** |
| **GPL-3.0** | 强 Copyleft，保护生态 | 阻碍闭源集成与部分平台分发策略 | 不采用 |
| **AGPL-3.0** | 网络服务也需开源 | 与「本地优先、可选云/AI」路线冲突 | 不采用 |
| **MPL-2.0** | 文件级 Copyleft | 多 crate 边界复杂，收益有限 | 不采用 |
| **MIT OR Apache-2.0**（双许可） | Rust 常见 | 对单一应用项目略冗余；Apache 已覆盖主要诉求 | 可选，当前不必 |

### 3. 与依赖栈的兼容性

- **Tauri 2**：上游为 MIT **或** Apache-2.0 双许可 → 与 Apache-2.0 兼容。
- **Rust 生态**（tokio、unicode-* 等）：多为 MIT / Apache-2.0 → 兼容。
- **libssh2**：BSD 风格 → 兼容；需在发行说明中保留其版权声明（随依赖传递）。

### 4. 未采用 Copyleft 的原因

终端是**系统级工具**，用户与厂商可能：

- 打包私有配置、企业 CA、内网插件；
- 在不变更核心的前提下做 OEM 发行。

GPL/AGPL 会迫使下游开源整个衍生桌面，不利于 adoption。我们选择在 **核心开源 + 文档透明** 上建立信任，而非用许可证锁定分发形态。

## 贡献与版权

- 贡献者保留著作权，通过 Pull Request 将贡献在 **Apache-2.0** 下授权给项目。
- 建议在首次较大 PR 时在描述中确认同意许可条款（见 PR 模板）。
- 若未来引入 **CLA**（Contributor License Agreement），将单独讨论并更新本文；当前阶段保持轻量。

## 安全与责任

- 安全报告流程见 [SECURITY.md](../SECURITY.md)。
- 软件按 **「AS IS」** 提供，不承担终端操作导致的系统损害；用户对其 shell、SSH 凭据与远程命令负责。
